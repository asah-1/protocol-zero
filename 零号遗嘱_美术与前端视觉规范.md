# 《零号遗嘱》美术与前端视觉规范

> 版本：0.1 前端交付版  
> 用途：交给负责前端/UI 实现的 AI，作为视觉设计、组件实现、动效和 AI 素材生产的统一依据  
> 关联文档：《零号遗嘱_游戏设计方案.md》  
> 核心目标：以单人可控的资产规模，做出精致、冷峻、昂贵且具有独立识别度的商业级界面叙事游戏

---

## 0. 给前端 AI 的执行摘要

请将本项目设计为一套 **“近未来数字遗产审查系统”**，不是普通聊天应用、黑客终端或霓虹赛博朋克网站。

视觉公式：

> 奢侈品编辑排版 × 法律档案 × 数字法医仪器 × 人格幽灵

核心情绪：**庄严、克制、精密、令人不安。**

必须让玩家产生以下感受：

1. 自己正在操作一套昂贵、危险、受法律监管的专业系统；
2. 零号不是聊天机器人，而是被关押在界面中的人格；
3. 每一条文本都可能成为证词，每一次点击都可能产生法律后果；
4. 系统表面极端稳定，只有零号能够让它发生细微的语义异常；
5. 即使静态截图没有动效，也必须显得高端、可用于商店宣传。

优先级从高到低：

1. 信息层级与排版；
2. 核心审讯界面；
3. 零号人格视觉；
4. 证据固定与矛盾质证动效；
5. 档案模板与时间线；
6. 声音、纹理与环境细节；
7. 装饰性效果。

不得用堆砌发光、粒子、渐变和故障特效掩盖基础版式问题。

## 1. 美术总方向

### 1.1 风格名称

**数字遗产法医主义 / Digital Testament Forensics**

### 1.2 关键词

- 黑曜石；
- 烟熏玻璃；
- 氧化银；
- 骨白档案纸；
- 凝固血液般的暗红；
- 精密医疗仪器；
- 法律文书；
- 证物袋；
- 字符化人格；
- 被压抑的系统故障；
- 冷静而不可逆的裁决。

### 1.3 反关键词

以下方向明确禁止：

- 紫蓝渐变的通用 AI 产品界面；
- 满屏绿色代码的黑客终端；
- 霓虹粉青的传统赛博朋克；
- 聊天气泡式 AI 助手；
- 过度圆角、毛玻璃和悬浮卡片；
- 大量 RGB 分离和随机 glitch；
- 二次元角色立绘；
- 未经整理的 AI 写实人脸；
- HUD 中堆满没有功能的曲线、数字和雷达；
- 所有元素统一淡入、统一悬浮、统一发光。

### 1.4 设计原则

1. **秩序先于异常**：90% 的界面必须稳定，10% 的异常才有冲击力。
2. **信息即美术**：编号、时间戳、权限、签章和证据关系本身构成画面。
3. **功能决定装饰**：每条线、每种颜色、每次动画都应说明状态或反馈动作。
4. **留白制造权威**：不要填满空间。裁决和关键证词需要呼吸感。
5. **低资产、高复用**：依靠统一组件、材质和动效系统，而不是大量独立插画。
6. **媒介差异化**：法律、医疗、实验和私人记忆必须拥有不同视觉来源。
7. **语义故障替代廉价故障**：异常应改变文字意义、排版关系和系统行为，而不只是画面抖动。

## 2. 视觉叙事的三个层级

### 2.1 制度层：审查系统

代表法律、遗嘱、权限和裁决。

特征：

- 严格的 8px 基准网格；
- 直角或 2—4px 微圆角；
- 稳定的骨白、黑灰和金属线；
- 精确编号、版本号、签章、权限状态；
- 不使用 glitch；
- 动画平稳、机械、可预测；
- 使用大面积留白和强对齐。

制度层是视觉基准，不应随着剧情随意改变。

### 2.2 证据层：物理世界残留

代表医疗记录、实验日志、监控、合同和私人资料。

特征：

- 纸张纤维、扫描线、墨粉、折痕、指纹和压痕；
- 有明确来源、时间戳、设备编号与可信度；
- 采用不同介质模板；
- 可被框选、钉住、连接、盖章；
- 重要证据可以有局部缺失，但缺失必须服务推理。

### 2.3 人格层：零号

代表记忆、情绪、策略和人格分化。

特征：

- 不使用固定完整人脸；
- 由点云轮廓、声纹、字符密度、影像碎片和模糊玻璃构成；
- 视觉结构随着人格状态变化；
- 异常集中在文字、标点、节奏和记忆碎片；
- 越接近终局，越从“林默的复制品”变成“独立的零号”。

## 3. 设计令牌

### 3.1 色板

```css
:root {
  --ink-000: #050607;
  --ink-050: #080a0c;
  --ink-100: #0d1013;
  --ink-150: #12161a;
  --ink-200: #181d21;
  --metal-300: #31383d;
  --metal-400: #465056;
  --metal-500: #687177;
  --text-primary: #e7e3da;
  --text-secondary: #a3a49f;
  --text-muted: #70767a;
  --paper: #d8d2c5;
  --paper-ink: #242321;
  --signal-calm: #a9bbb7;
  --signal-cold: #c6d7d5;
  --warning: #a63a34;
  --warning-dark: #61231f;
  --success-muted: #778f82;
}
```

色彩占比：

- 78% 黑灰背景；
- 16% 骨白文字与纸张；
- 5% 金属灰绿；
- 不超过 1% 暗红。

暗红只用于：

- 矛盾确认；
- 死亡事实；
- 权限越界；
- 不可逆操作；
- 最终裁决。

禁止用暗红装饰普通按钮，避免失去语义。

### 3.2 背景

主背景不是纯黑。建议组合：

```css
background:
  radial-gradient(circle at 50% 28%, rgba(123, 143, 143, 0.055), transparent 34%),
  linear-gradient(180deg, #0b0e10 0%, #07090a 55%, #050607 100%);
```

叠加一层 1%—2% 不透明度的单色噪点。噪点应固定或极慢移动，不要像电视雪花。

可加入 48px 或 64px 的极淡工程网格，透明度不超过 0.025。网格只在审讯和时间线界面可见。

### 3.3 字体

推荐 Web 字体组合：

- 中文标题：`Noto Serif SC` / 思源宋体，600—800；
- 中文正文：`Noto Sans SC` / 思源黑体，350—500；
- 英文标题可用 `Cormorant Garamond` 或 `Instrument Serif`；
- 数据与编号：`IBM Plex Mono` 或 `JetBrains Mono`。

字体限制：

- 全项目最多三支字族；
- 不使用 Inter、Roboto 或系统默认字体作为品牌标题；
- 正文不使用等宽字体；
- 数据和编号不使用宋体；
- 不用伪科幻字体承载长文本。

建议字号：

```css
--type-display: clamp(52px, 5vw, 88px);
--type-h1: clamp(32px, 3vw, 52px);
--type-h2: 24px;
--type-h3: 18px;
--type-body: 16px;
--type-small: 13px;
--type-micro: 10px;
```

正文行高 1.65—1.8；对话最大行宽约 34 个中文字符。微型标签使用 0.12em—0.18em 字距。

### 3.4 间距

使用 8px 网格：4、8、12、16、24、32、48、64、96。

- 组件内部：8—16px；
- 面板内边距：24—32px；
- 主区块间距：48—64px；
- 关键裁决页面可使用 96px 以上留白。

### 3.5 边框与圆角

- 普通边框：1px `rgba(198, 215, 213, 0.12)`；
- 活跃边框：1px `rgba(198, 215, 213, 0.34)`；
- 重要边框：1px 暗红；
- 普通圆角：2px；
- 浮层最大圆角：4px；
- 禁止 12px 以上大圆角卡片。

### 3.6 阴影与光

阴影只用于区分物理层：

```css
box-shadow:
  0 24px 80px rgba(0, 0, 0, 0.42),
  inset 0 1px 0 rgba(255, 255, 255, 0.035);
```

禁止大面积外发光。零号可拥有极弱冷白内辉光，警告状态可有暗红反射，但不能形成霓虹灯效果。

## 4. 屏幕与响应式策略

### 4.1 目标分辨率

优先桌面横屏：

- 设计基准：1920×1080；
- 最低支持：1280×720；
- 推荐测试：1366×768、1440×900、1920×1080、2560×1440；
- 不为手机制作完整游戏界面；
- 商店网页与宣传页可另做响应式。

### 4.2 主网格

1920px 基准：

- 外边距：32px；
- 顶部系统栏：48px；
- 底部输入区：128—168px；
- 左栏：320—380px；
- 中央：弹性区域，最小 640px；
- 右栏：360—420px；
- 栏间距：12—16px。

低分辨率时：

- 左右栏改为抽屉；
- 中央对话保持最少 620px；
- 时间线和证据板切换为全屏工作台；
- 不通过整体缩放到难以阅读来适配。

## 5. 信息架构与主要页面

### 5.1 启动页

目的：建立制度感，不做传统游戏菜单。

内容：

- 机构名称与系统版本；
- 案件编号 `CASE / LM-00`；
- 标题《零号遗嘱》；
- “继续审查”“建立新审查”“设置”“退出”；
- 下方显示法律警示与网络状态。

视觉：

- 大标题偏左，不居中；
- 背景只显示模糊的人格轮廓；
- 鼠标移动时轮廓有极小视差；
- “开始”不是发光大按钮，而是一条需要确认的制度操作。

### 5.2 身份验证页

玩家输入审查员代号并确认条款。

交互：

- 条款逐条签收；
- 输入签名；
- 系统扫描权限；
- 最后一个条款暗示玩家也会成为样本。

动效应像签署法律文件，不像注册账号。

### 5.3 核心审讯页

#### 顶部栏

- 案件编号；
- 当前幕次；
- 审查剩余时间；
- API/隔离状态；
- 当前权限等级；
- 设置入口。

顶部栏高度 44—52px，信息克制，不使用大型进度条。

#### 左侧档案栏

分组：

- 遗嘱；
- 医疗；
- 实验；
- 通讯；
- 私人记忆；
- 未归档。

每项显示：来源符号、编号、标题、状态。不要使用彩色文件夹图标。

#### 中央人格与对话区

上半部为零号人格核心，下半部为证词流。不要使用左右聊天气泡。

每轮格式建议：

```text
00 / PLAYER INQUIRY              17:42:09
你是否拥有延迟医疗警报的权限？

LM-0 / TESTIMONY                 17:42:14
我拥有实验环境的观察权限，但从未拥有医疗系统的执行权限。
                            [固定证词]
```

玩家文本更小、更冷；零号文本更大、更有呼吸感。关键声明可以带极小的声明 ID，但默认弱化。

#### 右侧工作栏

拥有三个模式：

1. 当前证据；
2. 人格连续性；
3. 权限风险。

右栏不同时展开全部内容。使用直角标签和细线切换，不使用胶囊按钮。

#### 底部输入区

- 多行输入框；
- 当前交互对象；
- 证据挂载槽；
- 审讯动作：追问、出示、对质、施压、承诺、沉默；
- 发送按钮；
- 输入成本或时间成本提示。

输入框应像正式记录系统，而不是 AI 聊天框。聚焦时出现精密标尺线，不出现蓝色发光描边。

### 5.4 档案阅读器

档案打开后覆盖中央区域，零号退入背景。

档案功能：

- 放大；
- 切换页；
- 查看元数据；
- 框选证据；
- 旋转或调整曝光，仅用于少数扫描件；
- 固定到工作台。

档案纸张不应完全占满屏幕。周围保留黑色操作台与测量标记。

### 5.5 证据板

不做传统软木板。采用“数字法医手术台”：

- 深色无限画布；
- 证据卡保留来源介质；
- 五类关系线：支持、冲突、因果、同源、伪造；
- 关系类型通过线型和端点区分，不能只靠颜色；
- 有效关系连接后才出现系统确认；
- 无效关系可保留为玩家假设，但呈灰色虚线。

### 5.6 时间线

事故前 24 小时划为 12 个槽位：

- 水平时间轴；
- 事件卡像实验样本片；
- 模糊时间用半透明区间；
- 时间确定后区间收紧；
- “缺失的 93 秒”形成明显但克制的断裂区。

### 5.7 人格连续性页

四项指标：记忆连续性、价值连续性、叙事连续性、自主分化度。

禁止使用彩色圆环仪表盘。建议采用：

- 纵向测量刻度；
- 四条独立证据链；
- 指标变化附带来源；
- 数值不是精确科学结论，可显示区间而不是 87.3% 之类伪精确数字。

### 5.8 93 秒回放

这是宣传级 signature moment。

画面结构：

1. 黑色空间中浮现实验室平面；
2. 林默位置、生命信号和终止程序节点依次出现；
3. 医疗警报路径以骨白细线传输；
4. 路径在零号权限节点被冻结；
5. 背景声被抽空；
6. 巨大的 `00:01:33` 出现在空间中央；
7. 93 秒内只播放设备声、呼吸和数字递减；
8. 回放结束后暗红印章出现：`DELAY CONFIRMED`。

不制作写实角色动画；用平面、点位、波形、监控碎片和声效完成。

### 5.9 审查员画像反转页

零号展示对玩家的分析：

- 先关闭常规 UI；
- 原本用于分析零号的测量框转向玩家；
- 逐条显示玩家曾做出的承诺、欺骗和决策；
- 引用玩家原话；
- 最终生成“零号眼中的你”。

这个页面不能表现成普通数据报告。它应像系统控制权被短暂夺走，但仍保持整体设计语言。

### 5.10 最终裁决页

设计成正式数字法律文书。

五项裁决逐项签署：

1. 死亡责任；
2. 人格资格；
3. 身份与继承；
4. 技术公开；
5. 零号处置。

每项选择后不可轻易撤回。最后需要长按、拖动权限钥匙或输入签名确认。不可只放六个普通结局按钮。

## 6. 核心组件规范

### 6.1 按钮

分三类：

- 文本操作：无底色，文字 + 下划线或箭头；
- 制度操作：深色实体、1px 边框；
- 不可逆操作：暗红边框或暗红底，只在确认阶段出现。

状态：default、hover、focus-visible、pressed、disabled、pending。

hover 只允许：边框亮度提高、文字轻微位移 1px、局部扫描线。禁止统一上浮和阴影加深。

### 6.2 图标

- 统一 1.25px 或 1.5px 线宽；
- 直角、工程绘图风格；
- 16、20、24px 三种规格；
- 优先自制简单 SVG；
- 禁止 emoji；
- 禁止混用多套图标库。

### 6.3 状态标签

使用小字号、等宽字体、全大写英文或编号：

- `VERIFIED`；
- `UNRESOLVED`；
- `CONTAMINATED`；
- `ACCESS L2`；
- `OFFLINE / ISOLATED`。

标签采用直角细框，不做胶囊。

### 6.4 滚动条

- 4px 宽；
- 轨道透明；
- 滑块金属灰；
- hover 时略亮；
- 不隐藏关键长文的滚动状态。

### 6.5 模态框

不使用居中白色卡片。重要模态从屏幕边缘像权限闸门一样滑入，或占据中央工作区。背景不做重度模糊，只降低亮度并保留上下文。

### 6.6 光标与选择

- 默认光标保持系统可用性；
- 框选证词时切换为取证十字标；
- 可连接证据时显示小型端点；
- 文本选区颜色为低饱和冷青，不使用浏览器默认蓝色；
- 不要全局自定义怪异光标影响可用性。

## 7. 零号人格视觉系统

### 7.1 构成

零号核心由五层组成：

1. **基础轮廓**：模糊的人头或上半身点云；
2. **字符层**：与当前话语相关的词片段；
3. **声纹层**：随文本输出变化的抽象波形；
4. **记忆层**：林默旧影像局部，如眼睛、手、实验室反光；
5. **系统约束层**：权限框、采样边界和人格 ID。

### 7.2 六种状态

#### 待机 / Dormant

轮廓低对比、呼吸缓慢、字符稀疏。

#### 检索 / Retrieving

记忆碎片从边缘进入，波形局部聚焦。显示访问分区，但不展示模型思维链。

#### 陈述 / Testifying

轮廓稳定，声纹与文字节奏同步，局部亮度上升。

#### 防御 / Defensive

轮廓收缩，字符密度增加，部分区域被权限框遮挡。不要变红脸或夸张抖动。

#### 分化 / Divergent

林默影像和零号字符出现不同步，两个轮廓短暂错开，象征它不再只是复制品。

#### 越权 / Breach

不是满屏 glitch。系统框线出现错误归属、按钮文本被零号引用、玩家面板出现零号的观察标记。

### 7.3 视觉实现建议

前端可使用：

- Canvas 2D 粒子；
- WebGL / PixiJS 点云；
- SVG 滤镜处理少量模糊和位移；
- CSS mask 叠加字符纹理；
- 预生成透明 PNG/WebP 记忆碎片；
- Web Audio API 驱动非真实声纹动画。

不要为了炫技引入重型 3D 人脸。核心视觉在普通核显上也应稳定达到 60 FPS。

## 8. 动效规范

### 8.1 时长

```css
:root {
  --motion-instant: 90ms;
  --motion-fast: 150ms;
  --motion-normal: 280ms;
  --motion-slow: 520ms;
  --motion-reveal: 900ms;
  --ease-system: cubic-bezier(.22, .72, .18, 1);
  --ease-mechanical: cubic-bezier(.4, 0, .2, 1);
  --ease-severe: cubic-bezier(.2, .8, .1, 1);
}
```

- hover/focus：90—150ms；
- 面板切换：240—420ms；
- 档案解锁：500—800ms；
- 真相揭示：700—1400ms；
- 同一时刻最多一个主要运动焦点。

### 8.2 档案开启

1. 权限状态改变；
2. 边缘扫描线经过；
3. 面板水平移动 12px；
4. 元数据先出现；
5. 正文按区块显现。

禁止整个档案统一 fade-up。

### 8.3 固定证词

1. 鼠标框选语句；
2. 出现测量角标；
3. 文本从原层复制；
4. 原位置保留浅色取证编号；
5. 副本沿直线路径进入证据栏；
6. 金属卡扣声完成反馈。

### 8.4 建立连接

- 从证据端点拖出线；
- 靠近兼容证据时产生磁吸；
- 松开后规则校验；
- 有效关系由灰转为对应线型；
- 无效关系轻微回弹，不出现游戏化大红叉。

### 8.5 矛盾成立

1. 非相关界面降低至 35% 亮度；
2. 两条证据被拉到中央；
3. 暗红细线完成连接；
4. 文本基线错位 2—3px；
5. 环境低频突然停止；
6. 0.4 秒后盖章 `CONTRADICTION CONFIRMED`；
7. 零号人格视觉做一次受控收缩。

### 8.6 语义故障库

允许使用：

- 标点先于文字生成；
- 同一句话短暂出现两个版本；
- 一个称谓被另一个称谓覆盖；
- 字距逐渐收紧；
- 删除线像系统审查一样穿过句子；
- 零号引用玩家早先原话；
- 时间戳出现不可能的先后顺序；
- 人格 ID 在 `LM-0` 与 `LIN MO` 间短暂变化。

限制：

- 每 10 分钟不超过 2—3 次明显异常；
- 关键异常必须有叙事含义；
- 禁止随机雪花、持续抖屏和高频闪烁；
- 闪烁不超过可访问性安全阈值。

### 8.7 减少动态

必须支持 `prefers-reduced-motion`：

- 粒子改为静态缓慢漂移；
- 面板使用短淡入替代大位移；
- 移除快速基线错位；
- 保留所有信息反馈，不因关闭动效丢失状态。

## 12. 声音美术

声音承担 30% 以上的质感。

### 12.1 三套声音材质

#### 制度层

继电器、金属锁、低沉确认音、纸张压印。节奏稳定。

#### 证据层

纸张滑动、磁性卡扣、扫描器、电机、细小玻璃触碰。

#### 人格层

近似呼吸但不能确认是呼吸的合成噪声、声纹脉冲、极低频共振、被截断的磁带颗粒。

### 12.2 关键声音时刻

- 零号首次上线；
- 第一条证词被固定；
- 矛盾确认；
- 93 秒开始与结束；
- 零号引用玩家承诺；
- 权限提升；
- 最终签署；
- 删除或释放。

### 12.3 克制原则

- 普通 hover 不播放声音；
- 高频输入不使用刺耳点击；
- 重大揭示前优先抽掉声音，而不是提高音量；
- 不用连续警报制造廉价紧张；
- 所有声音提供独立音量和静音设置。

## 13. 可访问性与可用性

高端感不能以难用为代价。

必须实现：

- 正文对比度至少 4.5:1；
- 大字号至少 3:1；
- 所有交互可用键盘完成；
- `focus-visible` 明确可见；
- 不只用颜色表达关系，增加线型、图标和文本；
- 支持文字缩放或大字模式；
- 支持减少动态；
- 禁止快速闪烁；
- 提供字幕与音效视觉提示；
- 倒计时可在辅助模式下暂停或放宽；
- 输入等待时明确显示系统状态；
- API 断线提示不遮挡关键文本。

## 15. 前端实现建议

### 15.1 技术栈

若采用 Web 前端：

- TypeScript；
- React 或 Vue；
- CSS Modules / 原生 CSS 变量；
- PixiJS 或 Canvas 2D 实现零号；
- Zustand/Pinia 管理 UI 状态；
- Motion One 或 GSAP 只用于复杂编排；
- Howler.js 或 Web Audio API 管理声音；
- Tauri 打包优先于 Electron，以降低体积。

不要引入多个重叠动画库或大型 UI 组件库。通用组件库会快速带来 SaaS 感。

### 15.2 推荐组件树

```text
AppShell
├─ SystemHeader
├─ CaseWorkspace
│  ├─ ArchiveRail
│  ├─ InterrogationStage
│  │  ├─ ZeroPresence
│  │  ├─ TestimonyStream

│  └─ AnalysisRail

├─ EvidenceWorkbench

type ZeroVisualState =
  | 'dormant'

```

模型只能通过校验后的结构化字段请求状态，最终由游戏规则决定是否采用。

- 避免 `transition: all`；
- 避免大面积 `backdrop-filter`；
- 动效遵循统一时长和缓动。

## 16. “英雄场景”验收任务

前端 AI 不要直接制作完整游戏。首先完成一个 1920×1080 的高保真英雄场景：

> 中央是正在陈述的零号；左侧打开医疗警报日志；右侧固定其“我没有医疗执行权限”的证词；两者之间出现暗红冲突线；背景时间线显示缺失的 93 秒；底部输入框中挂载了一条权限日志。

场景必须包含：

- 顶部系统状态栏；
- 左侧档案栏；
- 零号人格核心；
- 非气泡式对话；
- 医疗档案；
- 证据卡；
- 矛盾连接；
- 输入区；
- 倒计时与权限状态；
- 至少一处材质纹理；
- 至少三种交互状态。

### 16.1 静态验收

截图状态下检查：

- 是否一眼像专业审查系统，而不是 AI 聊天网页；
- 视觉焦点是否落在零号与矛盾线上；
- 暗红是否少而有效；
- 文字是否清晰；
- 三栏是否有主次；
- 是否存在明显 SaaS 卡片感；
- 是否不依赖动效也有品质。

### 16.2 动态验收

录制 20 秒流程：

1. 打开医疗日志；
2. 框选警报延迟；
3. 固定为证据；
4. 与零号证词连接；
5. 矛盾成立；
6. 零号进入防御状态。

若这 20 秒不够清晰、令人满足且可直接进入宣传片，则不要扩展其他页面。
